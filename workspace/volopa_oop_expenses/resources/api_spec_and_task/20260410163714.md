## Shared Knowledge

The system uses Laravel's built-in OAuth2 middleware (Oauth2UserClient) applied at route group level for authentication. All data operations are scoped by client_id for multi-tenancy. The system follows flag-based soft delete pattern with 'deleted' TINYINT(1) and 'delete_time' DATETIME columns. Timestamp columns use 'create_time' and 'update_time' instead of Laravel's default timestamps. Foreign key relationships are explicitly defined with constraints. The expense workflow follows draft->submitted->approved/rejected status flow. CSV uploads are processed asynchronously using Laravel queues with the ProcessExpenseUpload job. FX conversion integrates with existing platform /wallet-ccy-value endpoint. All API responses use API Resources for consistent formatting and field hiding.

## Anything UNCLEAR

Several key implementation details require clarification: 1) The notification mechanism for batch upload completion (in-app, email, push, or combination) affects the ProcessExpenseUpload job implementation. 2) The ClientFeatures model structure and query pattern for checking if client has OOP Expense feature enabled (feature_id = 16) is needed for proper authorization. 3) The audit logging implementation for permission grants/revokes - whether to use existing platform audit infrastructure or create project-specific audit tables. 4) Consistency in soft delete convention - whether to use flag-based (deleted + delete_time) or Laravel SoftDeletes (deleted_at) pattern for new tables, as both appear in specifications.
