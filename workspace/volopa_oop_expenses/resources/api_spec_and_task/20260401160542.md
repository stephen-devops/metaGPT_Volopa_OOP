## Shared Knowledge

All models use Volopa legacy timestamp pattern (create_time, update_time) and flag-based soft delete (deleted, delete_time). The existing Oauth2UserClient middleware handles authentication at route group level. FX conversion service integrates with existing platform FX infrastructure using 30-day lookback and commission calculation. CSV validation uses all-or-nothing approach with synchronous validation followed by background processing. User permission system supports role-based access with delegation capabilities where Admins can grant access to their managed users. All data operations are scoped by client_id for multi-tenancy. Standard Laravel patterns: Form Requests for validation, Policies for authorization, API Resources for response transformation, Services for business logic.

## Anything UNCLEAR

Four project-level decisions need clarification: 1) Notification channel preference for batch upload completion (in-app notification, email, push notification, or combination) - this affects the ProcessExpenseUpload job implementation but doesn't block core development. 2) Whether the FX conversion with wallet-ccy-value endpoint + 30-day lookback + commission calculation is reusing an existing platform service or requires new integration setup - this is referenced as existing infrastructure but the specific debounce + lookback + commission pattern may be new. 3) Audit logging mechanism for permission grants/revokes - whether to use a platform-standard audit system, separate audit table, event sourcing, or Laravel activity log. 4) Exact ClientFeatures model query pattern to check if a client has feature_id = 16 (OOP Expense) enabled - the model exists but the backing table and query structure need confirmation. These decisions affect implementation details but core API development can proceed with reasonable assumptions.
