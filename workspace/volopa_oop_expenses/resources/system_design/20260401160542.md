## Implementation approach

We will build a Laravel 10+ API system with three core modules: User Management & Permissions (RBAC with delegation), CSV Batch Upload (sync validation + async processing), and Single Expense CRUD (with FX conversion). The system follows Laravel best practices using Form Requests for validation, Policies for authorization, API Resources for response shaping, and Services for business logic. We'll use existing Volopa infrastructure (OAuth2 middleware, FX service, queue system) and implement multi-tenant architecture with client_id scoping. The CSV upload uses synchronous validation with atomic all-or-nothing processing, then background jobs for main service sync. FX conversion provides real-time currency conversion with user override capability. All data uses soft deletes for audit trails and follows the existing Volopa timestamp patterns.

    PUC->>CSV: validate(filePath, targetUserId, clientId, adminId)
    CSV->>CSV: preloadReferenceData(clientId)
    CSV->>CSV: validateHeaders(headers)
    CSV->>CSV: validateRow(row, lineNumber) for each row
    CSV-->>PUC: validation result
    alt validation failed
        PUC->>DB: update upload (status=validation_failed, errors)
        PUC-->>C: 422 validation errors
    else validation passed
        PUC->>DB: bulk insert to PocketExpenseUploadsData (status=pending)
        PUC->>DB: update upload (status=processing)
        PUC->>Queue: dispatch ProcessExpenseUpload(uploadId)
        PUC-->>C: 200 success response
        Queue->>Job: handle()
        Job->>Job: syncExpenseBatch() in batches of 100
        Job->>DB: update upload (status=completed)
    end


## System constraints

- All queries and mutations must be scoped by client_id for multi-tenancy
- user_id must match authenticated user (server-side check)
- Target entities (e.g. expense_user_id) must belong to the same client_id
- All API endpoints require OAuth2 authentication via Oauth2UserClient middleware
- Controllers MUST NOT call $this->middleware() in constructors — Oauth2UserClient is applied at the route group level only
- CSV files maximum 200 rows per file
- CSV file format: CSV or TXT with CSV content only
- CSV file size: max 10MB (10240 KB)
- CSV header row mandatory and must exactly match column names
- All-or-nothing validation: if any CSV row fails validation, no expenses are created
- Date must not be older than 3 years
- Date format: DD/MM/YYYY (DD-MM-YYYY in template)
- Currency Code must be 3-letter ISO and in allowed list
- Amount sign determined by expense type (Refund = positive, others = negative)
- VAT % must be numeric between 0-100 (strip % sign)
- Source must match configured sources for client (including global Other)
- Source Note required when Source = Other
- Merchant Name max length per DB definition (VARCHAR 180)
- Notes: trim and prevent SQL injection, follow DB limit
- Only Primary Admin has full access to all users by default
- Admin gets full access only to own expenses by default; needs explicit grant for others
- Business User and Card User cannot approve expenses even with management rights
- Managing access can be given to any user irrespective of role
- Admin can only grant access to their own managed users (not all users)
- Maximum 20 active expense sources per client
- Unique source names per client (client_id, name)
- Global 'Other' record (client_id = NULL) is not deletable or editable
- Soft-deleted sources excluded from future dropdowns but visible on historical records
- 3 defaults auto-created on feature enable: Cash, Corporate Card, Personal Card
- FX rate lookup: max 30-day lookback from expense date
- If no FX rate found, return 'No FX Available'
- Backend must recalculate FX on save (do not trust frontend-only value)
- Commission formula: AdjustedRate = BaseRate x (1 - Comm%)
- Tables with similar columns across modules are DISTINCT entities with different column types, status enums, and relationships. Each MUST have its own Model and Migration and Factory. Do NOT merge or conflate tables from different modules
- When tables track both a target user and the performing admin, DB column names and API form field names may differ. Refer to database_tables and api_routes for exact column-to-field mappings. Do NOT assume DB column names match API form field names
- Don't use a class or a method that doesn't exist in the current repository or was not defined in the added code
- Don't add methods that already exist in the current repository
- Don't return raw Eloquent models from API endpoints
- Don't use session/redirect patterns in APIs
- Don't return 200 for errors
- Don't respond with inconsistent shapes/casing
- Don't expose stack traces or sensitive error details
- Don't disable mass-assignment protection or trust client-owned fields
- Don't build query filters directly from user input
- Don't leak environment or config
- Don't create N+1 queries (especially in Resources)
- Don't return unbounded lists
- Don't ignore caching
- Don't let file uploads bloat the API process
- Don't forget transactions for multi-write operations
- Don't hardcode timestamps/timezones
- Don't forget observability
- InnoDB engine required for all new tables
- Character set: utf8mb4 for all new tables
- Collation: utf8mb4_unicode_ci for all new tables
- Flag-based soft delete columns: deleted TINYINT(1) DEFAULT 0, delete_time DATETIME DEFAULT NULL. Set deleted = 1 and delete_time = NOW()
- Volopa legacy timestamp columns: create_time DATETIME DEFAULT CURRENT_TIMESTAMP, update_time DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP
- user_feature_permission table uses unique constraint: unique_user_feature: (user_id, client_id, feature_id)
- pocket_expense_source_client_config table uses unique constraint: oop_expense_source_client_name_unique: (client_id, name)
- pocket_expense_metadata table uses unique constraint: unique_expense_metadata_type: (pocket_expense_id, metadata_type, deleted)
- pocket_expense_uploads_data table uses index: idx_upload_status: ['upload_id', 'status'], idx_pending_sync: ['status']
- opt_pocket_expense_type seed data: {'option': 'ATM Withdrawal', 'amount_sign': 'negative'}, {'option': 'Point of Sale', 'amount_sign': 'negative'}, {'option': 'Fee & Charges', 'amount_sign': 'negative'}, {'option': 'Refund from Merchant', 'amount_sign': 'positive'}
- pocket_expense status ENUM values: 'draft','submitted','approved','rejected' with DEFAULT 'draft'
- pocket_expense_metadata metadata_type ENUM values: 'category','tracking_code_type_1','tracking_code_type_2','project','additional_field','file','expense_source'
- pocket_expense_file_uploads status ENUM values: 'uploaded','validation_failed','validation_passed','processing','completed','failed','sync_failed' with DEFAULT 'uploaded'
- pocket_expense_uploads_data status ENUM values: 'pending','processing','synced','failed' with DEFAULT 'pending'
- Return proper HTTP status codes: 200 OK for retrieval, 201 Created for creation, 204 No Content for deletion, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 422 Unprocessable Entity for validation failed
- All foreign key relationships must reference existing tables: users(id), clients(id), features(id), transaction_category(id), tracking_codes(id), configurable_projects(id), file_store(id), expense_additional_field(id), opt_file_type(id)
- CSV upload API endpoint: POST /api/uploads/pocket-expense/csv (omits /v1 prefix per specification)
- CSV form fields: file (required|file|mimes:csv,txt|max:10240), user_id (required|exists:users,id), expense_user_id (required|exists:users,id), client_id (required|exists:clients,id)
- CSV success response: 200 {success:true, message, upload_id, total_rows}
- CSV error response: 422 {success:false, message, upload_id, total_rows, error_count, errors:[{line_number, field, error, value}]}
- ProcessExpenseUpload job syncs to main service in batches of 100 rows
- Background job queue name: 'expense-processing'
- CSV storage path pattern: storage/app/pocket-expense-uploads
- Do NOT build OopExpense model, OopExpenseController, OopExpenseService, OopExpensePolicy, OopExpenseResource, StoreOopExpenseRequest, UpdateOopExpenseRequest, or create_oop_expenses_table migration as pocket_expense supersedes oop_expenses
- pocket_expense is the authoritative expense table, not oop_expenses
- pocket_expense status values ('draft','submitted','approved','rejected') are authoritative over oop_expenses status values

## Anything UNCLEAR

These decisions affect implementation details but core API development can proceed with reasonable assumptions.
