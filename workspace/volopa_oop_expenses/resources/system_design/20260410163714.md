## Implementation approach

We will build a comprehensive Laravel 10+ API system for Volopa's Out-of-Pocket Expenses with three interconnected modules. The system leverages Laravel's built-in features including Eloquent ORM, Form Request validation, API Resources, Queue jobs, and Policy-based authorization. We'll use existing Volopa infrastructure (OAuth2 middleware, FX services, file storage) and follow the platform's multi-tenant architecture with client_id scoping. The implementation follows a service-oriented approach with thin controllers, comprehensive validation, and proper transaction management. Key components include: 1) RBAC system using user_feature_permission table with delegation hierarchy, 2) CSV batch upload with synchronous validation and asynchronous processing via Laravel queues, 3) Single expense CRUD with real-time FX conversion and flexible metadata storage. We'll use Laravel's migration system for database schema, factories for testing, and API Resources for consistent response formatting. The system integrates with existing platform services including wallet currency lookup, FX conversion, and notification infrastructure.

## System constraints

- All API endpoints must use Oauth2UserClient middleware applied at route group level via Route::middleware(['Oauth2UserClient'])
- Controllers must NOT call $this->middleware() in constructors - Oauth2UserClient is applied at the route group level only
- All queries and mutations must be scoped by client_id for multi-tenancy
- user_id must match authenticated user (server-side check)
- Target entities (e.g. expense_user_id) must belong to the same client_id
- All tables must use InnoDB engine with utf8mb4 character set and utf8mb4_unicode_ci collation
- New tables should use flag-based soft delete with deleted TINYINT(1) DEFAULT 0 and delete_time DATETIME DEFAULT NULL columns
- Timestamp pattern for new tables: create_time DATETIME DEFAULT CURRENT_TIMESTAMP and update_time DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP
- Maximum 200 rows per CSV file upload
- CSV file format must be CSV or TXT with CSV content, maximum 10MB (10240 KB)
- CSV header row is mandatory and must exactly match column names
- All-or-nothing validation: if any CSV row fails validation, no expenses are created
- Date must not be older than 3 years from current date
- Date format in CSV: DD/MM/YYYY (DD-MM-YYYY in template)
- Currency Code must be 3-letter ISO and in platform allowed list
- Amount sign determined by expense type (Refund = positive, others = negative)
- VAT % must be numeric between 0-100 (strip % sign from input)
- Source must match configured sources for client (including global Other)
- Source Note required when Source = Other
- Merchant Name maximum length VARCHAR(180) per database definition
- Maximum 20 active expense sources per client
- Unique source names per client via unique constraint (client_id, name)
- Global 'Other' record (client_id = NULL) is not deletable or editable
- FX rate lookup: maximum 30-day lookback from expense date
- If no FX rate found, return 'No FX Available'
- Backend must recalculate FX on save (do not trust frontend-only value)
- Commission formula: AdjustedRate = BaseRate x (1 - Comm%)
- 2-second debounce before triggering FX API call in frontend
- Only Primary Admin has full access to all users by default
- Admin gets full access only to own expenses by default; needs explicit grant for others
- Business User and Card User cannot approve expenses even with management rights
- Admin can only grant access to their own managed users (not all users)
- Tables with similar columns across modules are DISTINCT entities with different column types, status enums, and relationships
- Do not use a class or method that doesn't exist in the current repository or was not defined in the added code

- Do not return raw Eloquent models from API endpoints
- Do not use session/redirect patterns in APIs

- Do not expose stack traces or sensitive error details in API responses
- Do not disable mass-assignment protection or trust client-owned fields
